### Session lifecycle — formal definition

A session is a bounded unit of Kvothe's work, from startup-gate pass to shutdown-gate pass.

**Cross-midnight rule:** A unit (session or Kilvin spawn) belongs to its **open date** — the date embedded in the session-open or spawn marker at creation time. If a session opens on April 1st and closes on April 2nd, it belongs to April 1st. All checks (reflection file, marker lookups, watchdog) use the open date, not the close date. Implementation note: all file paths and marker names use `{open-date}` (the locked date from `.preflight-date`) or `{open-date-yesterday}` — never a wall-clock-relative placeholder like `{today}` or `{yesterday}`.

**Open-date lock:** The session open-date is computed **once, at the start of Stage 1 preflight**, and held fixed for the entire startup sequence. All Stage 1 file reads, chronicle creation, and the Stage 2 gate all use this precomputed open-date. If the clock crosses midnight during Stage 1 or Stage 2, the locked open-date does not change. The session-open marker written by Stage 2 uses the same locked date. This prevents Stage 1 writing to the April 1 chronicle while Stage 2 writes an April 2 session-open marker.

**`{open-date-yesterday}` defined:** Throughout this spec, wherever startup or gate steps reference "yesterday," that means `{locked-open-date} minus one calendar day` — not the wall-clock date at the moment each step runs. The term `{open-date-yesterday}` is used in all startup and gate paths to make this explicit. Its value is fixed at Stage 1 start alongside the locked open-date.

**Session open:**
When startup-gate.sh passes, it writes a session-open marker using the exact `session_candidate` value from `.preflight-date`:
```
memory/synapse/.session-open-{session_candidate}
```
Where `{session_candidate}` is `{open-date}-{HHMM}` as stored in `.preflight-date` — not a freshly computed value at pass time.
Content: `{ "opened_at": "{ISO timestamp}", "gate_version": "2.0" }`

**Session close:**
When shutdown-gate.sh passes, it writes a session-close entry to session-log.md (see below).
The `.session-open-{session_candidate}` marker is deleted on clean close.

**Multiple sessions per day:**
Each session has its own timestamped open marker. Session-log gets one entry per session. Reflections are appended to the same daily file.

**Watchdog check:**
If a `.session-open-{session_candidate}` marker exists but no session-log entry with `Session ID: {session_candidate}` is present → session is active or was not properly closed. **Matching is by Session ID** (the `session_candidate` value embedded in the marker filename), not merely by date. This ensures two sessions opening on the same day are tracked independently — a close entry for one cannot satisfy the check for the other. Watchdog alerts after **4 hours of inactivity** — defined as no observable workspace writes to `ACTIVE.md`, `memory/reflections/`, or `memory/guidance/` within the last 4 hours. These are the three paths in the alert set. Exempt (never count toward inactivity reset): writes to `synapse/proposals/*.reviewed` and `synapse/contradiction-log.md` — valid preflight artifacts. No other paths in the alert set. A session that is actively writing to the alert-set paths is not flagged regardless of how long it has been open. The alert means "no recent activity in alert-set paths AND no close entry matching this Session ID" — a prompt for manual verification, not evidence the session failed.

### Session compliance log format

**File:** `memory/synapse/session-log.md`
**Model:** Append. One entry per session close. Two valid entry forms: (1) normal close — `Session type: active | heartbeat-only` with real field values; (2) manual close — `Session type: unknown` with `unknown` field values, written during orphan resolution (see CHECK 7 for format). Both forms are valid schema entries; parsers must accept `unknown` as a field value.
**Heading date:** The `{open-date}` in `## Session — {open-date} — {HH:MM} close` is the session's **open-date** (from `session_candidate`), not the wall-clock date at close time. A session that opens April 1 and closes April 2 is logged as `## Session — 2026-04-01 — ...`. The authoritative key is always `Session ID: {session_candidate}`; the heading date is for human readability and grouping only.

```markdown
---
## Session — {open-date} — {HH:MM} close
Session ID: {session_candidate}
Session type: active | heartbeat-only
- Archive review (GENERAL_LEARNINGS): [trigger fired → G{nn} written | no trigger fired]
- Archive review (NEEMCLAW_LEARNINGS): [trigger fired → N{nn} written | no trigger fired]
- Kvothe reflection: [written {HH:MM} | heartbeat-only — skipped]
- Proposals reviewed: [{N} candidates disposed | no proposals | deferred from prior session]
- Kilvin spawns (this session): [{N} spawns, {N} reflections verified | no spawns this session | {N} spawns, {M} reflections — {N-M} backup entries written | no spawns this session — {N} backup entries written for earlier same-day spawns (open-date cumulative enforcement gap)]
```

Watchdog reads this file to find close entries by **Session ID** — the `Session ID:` field in each entry. For each `.session-open-{session_candidate}` marker, watchdog checks whether a session-log entry with `Session ID: {session_candidate}` exists. No match = unresolved session. Does NOT check archive file modification dates. Does NOT match by date alone — open-date is used only for grouping and reporting; the authoritative key is always the full `session_candidate`.

**GENERAL_LEARNINGS.md — write when:**
1. A mistake caused measurable rework, cost, or wasted time
2. Rai corrected something — any correction
3. A discovery about how a system actually works (differed from assumption)
4. A pattern repeating across 3+ sessions
5. A new protocol, rule, or constraint Rai defines

1. A bug was caught in QA that a spec should have prevented
2. A build failure caused by a wrong assumption about the codebase

5. A Kilvin timeout or failure pattern emerged

**If no trigger fires:** write nothing to archives. Record in session-log only.

### Reflection quality

Word count (<20 words per question) is the minimum floor — labeled as a weak signal, not a quality guarantee.

The primary quality mechanism is synthesis signal degradation: empty or padded reflections produce no patterns, no personal guidance is built. That is the natural consequence.

Synthesis flags low-quality reflections in the checkpoint. The 30% threshold is applied only when the agent has **≥5 reflections in the 7-day window** (minimum sample size). Below 5, rate is noted but no intervention triggered. Thresholds are tunable policy defaults, not proven optimal values.

### Contradiction handling — two-level standard

**Level 1 — Open a contradiction:**
- 1 strong tier 1 or tier 2 data point that conflicts with agent's stated belief in a reflection
- Confidence: any level ≥ 0.70
- Result: contradiction entry added to contradiction-log.md with status OPEN

**Level 2 — Act on a contradiction (change personal guidance):**
- 3 independent tier 1 or tier 2 data points confirming the same conflict
- OR 1 tier 1/2 data point with confidence ≥ 0.90
- Result: Kvothe promotes updated guidance to personal index

**What tier 3 alone can do:**
Create a "pattern worth examining" proposal. Cannot create a formal contradiction entry.

**Contradiction-log entry format:**
```markdown
## Contradiction #{N} — {agent} — {date}
- Existing guidance: {P-key} — "{current rule}"
- Agent stated: "{from reflection}"
- Conflicting evidence: {source, tier, reference}
- Level 1 met: {yes — date opened}
- Level 2 met: {yes — date | no — N/3 confirmations}
- Status: OPEN | RESOLVED | SUPERSEDED | INVALID
- review_note: {Kvothe's decision at startup review}
- Resolution: {what changed}
- Resolved at: {date}
```

**Status transitions:**

| From | To | Condition |
|---|---|---|
| OPEN | RESOLVED | Level 2 met + Kvothe updated guidance |
| OPEN | SUPERSEDED | Newer evidence changed the basis |
| OPEN | INVALID | Kvothe determined evidence was misinterpreted |
| OPEN (with review_note) | OPEN | Acknowledged, deferred — passes startup gate |

### Promotion ladder

This is the normal path. One explicit exception exists: direct-to-shared from strong tier-1 multi-agent chronicle evidence — defined below Step 4. That exception is manual Kvothe-only and never generated by synthesis.

```
STEP 1 — REFLECTION
Agent writes reflection (self-report, tier 3).

STEP 2 — PROPOSAL CANDIDATE
4AM synthesis generates candidate when:
  - Same pattern in 3+ reflections from this agent
  - OR tier 1/2 evidence confirms reflection's claim (one data point sufficient to open)
Candidate tagged: agent, evidence_tier, confidence, proposed_key

STEP 3 — PERSONAL GUIDANCE
Kvothe promotes to personal index when:
  - Confidence ≥ 0.85 AND
  - Tier 1/2 present (or 3+ independent reflections), AND
  - Not a semantic duplicate of existing entry
Written to: memory/guidance/{agent}.md as P-{agent}-{nn}

STEP 4 — SHARED GUIDANCE
Candidate for shared when:
  - Same pattern independently appears in personal guidance for 2+ different agents
  - Personal entries must already exist (promoted in Step 3)
  - Synthesis flags the cross-agent match; Kvothe reviews and promotes
Written to: memory/guidance/shared.md as GS-{nn} with confirmed-by list
```

**Exception (direct to shared):** A pattern may skip Step 3 and go directly to GS- if tier 1 chronicle evidence shows the same failure mode affecting multiple agents in a single documented event. Exception must be explicitly noted in the GS entry with the evidence reference.

## Part 8: Synthesis pipeline (4AM cron)

**Schedule:** Daily 4:00 AM Manila. Isolated Sonnet 4.6 session. Unchanged.

**Reads and catch-up contract:**

Two persistent state files govern what synthesis reads:

```
memory/synapse/file-mtimes.json       ← records last-processed mtime per reflection file
memory/synapse/processed-registry.jsonl  ← global log of all processed entry IDs (append-only)
```

On each synthesis run, the cron:

**Step A — Determine what to read:**
1. List all files matching `memory/reflections/*/*.md`
2. For each file, compare its current mtime against `file-mtimes.json`
3. If file is new (not in file-mtimes.json) or modified since last recorded mtime: add to read list
4. For each reflection file added to the read list, extract its open-date from the filename (`YYYY-MM-DD.md`). Add `memory/chronicles/{open-date}.md` to the chronicle read list (if it exists). This ensures synthesis always has the matching chronicle context for any reflection being processed — including late-arriving entries from older open-dates.
5. Also always read `memory/chronicles/{run-date-yesterday}.md` (primary chronicle for this run)
6. Also always read `memory/guidance/{agent}.md` (all 11) and `memory/guidance/shared.md`

**Chronicle coupling rule:** A reflection entry may only be compared against evidence from its own open-date chronicle. Synthesis must not cross-contaminate — an April 1 late entry is evaluated against the April 1 chronicle, never against April 2 or April 3.

**Step B — Dedupe within read list:**

**Canonical dedupe ID (single authoritative definition):**
```
{agent}::{open-date}::{N}::{HH:MM}
```

Where `{N}` is the per-file append-index (1 for the first entry in the file, 2 for the second, etc.). Agents count existing `## Reflection —` entries in the file before appending to determine N. No two entries in the same file can share the same N — the ID is collision-proof.

**Heading format (all four reflection types):**
```
## Reflection — {HH:MM} — {N} — {situation or task label}
```

The situation/task label is for human readability and pattern naming only — it is not part of the ID.

**Registry check:** Before processing any entry, check `processed-registry.jsonl` for its 4-tuple ID. If found: skip. If not found: process.

**Step C — Process and record (commit order):**
1. Process all new/unprocessed entries (per-agent analysis, cross-agent analysis, generate proposals)
2. Write proposals to `memory/synapse/proposals/{run-date}.md` and synthesis narrative to `memory/synapse/chronicle-reflections/{run-date}.md`
3. Update contradiction-log — **idempotent:** before opening a new entry, check if an OPEN entry already exists matching the same **agent + conflicting guidance key + evidence reference**. If yes: skip. If no: append new entry. Using all three fields means a single evidence reference can legitimately open multiple contradiction entries for the same agent if it conflicts with different guidance keys — but will not open the same contradiction twice.
4. **Only after steps 2–3 succeed:** append newly-processed entry IDs to `processed-registry.jsonl`
5. **Only after step 4 succeeds:** update `file-mtimes.json` with current mtime of each file read
6. Write checkpoint to `memory/synapse/checkpoint-{run-date}.md` (metadata only)

**Commit order rationale:** If the run crashes between steps 1 and 4, the IDs are not in `processed-registry.jsonl`. The next run will reprocess those entries — duplicate proposals are acceptable and recoverable (Kvothe discards at review); contradiction-log is idempotent (step 3) so a reprocessed entry checks for an existing OPEN entry with the same agent + guidance key + evidence reference and skips rather than duplicating. The inverse failure (IDs written but proposals not materialized) causes silent data loss and is not recoverable. Write outputs first, registry second.

**Reflection entry immutability rule:**
Once a reflection entry has been processed (its ID appears in `processed-registry.jsonl`), that entry is immutable. If the agent needs to correct or amend it, they must append a new entry with a new timestamp — never edit the existing heading or body. Editing an existing processed entry will not trigger reprocessing — the 4-tuple ID will match the registry and synthesis will skip it. Agents are responsible for knowing this rule. It must be stated in their SOUL.md under the Synapse section.

**Why this works for late entries:**

An April 1 session that closes at 5AM April 2 appends its reflection to `reflections/{agent}/2026-04-01.md`. The April 2 4AM run has already completed. On April 3, Step A compares the current mtime of `2026-04-01.md` against `file-mtimes.json`. The mtime is newer than the recorded value — the file was modified after the last run read it. April 3 synthesis adds the file to its read list, processes the new entry, and its 4-tuple ID is not in `processed-registry.jsonl` so it is processed. Entry captured. Maximum delay: one synthesis cycle. No entry is permanently missed as long as the session eventually closes.

**Checkpoint ownership (unambiguous):**
- `processed-registry.jsonl` is global and append-only. All processed entry IDs from all runs accumulate here. Dedupe always consults this single file.
- `file-mtimes.json` is a key-value map. Each synthesis run updates mtime values for files it read.
- `checkpoint-{run-date}.md` contains run metadata only — no entry IDs. One per run date.
- There is no per-open-date checkpoint registry. The registry is global.

**Processing:**

1. Per-agent: classify evidence tier, compare against chronicle + personal guidance, generate candidates
2. Cross-agent: flag same pattern in 2+ agents as shared candidate (requires personal entries to exist for normal path). Exception path — direct-to-shared from strong tier-1 chronicle evidence — is a **manual Kvothe-only promotion**. Synthesis does not generate exception-path GS candidates. If Kvothe identifies a multi-agent tier-1 event warranting a direct GS entry, he creates it manually and marks `path="exception"` in shared.md.
3. Write proposals to `memory/synapse/proposals/{date}.md` with evidence_tier, confidence, promotion_path
4. Update contradiction-log (Level 1 check: 1 strong tier 1/2 conflict → open entry)
5. Write checkpoint with: agents processed, agents missing, low-quality reflections (flagged, not threshold-applied), proposal counts
6. Send system event to Kvothe

**Proposal format:**
```
## Candidate {N}
- agent: {name or "fleet"}
- type: personal | shared-candidate | contradiction-open | pattern-worth-examining
- proposed_key: P-{agent}-{nn} or GS-{nn}
- proposed_rule: {one-line rule}
- evidence_tier: {1|2|3|4}
- evidence: {reference}
- confidence: {0.70–1.00}
- promotion_path: requires_personal_first: yes (synthesis only generates normal-path candidates; exception path is manual Kvothe-only)
- related_guidance: {existing key if any}
```

**Synthesis note:** For agents without chronicles (Lesley, Amina, Boots, Auri when not chronicle-keeping), synthesis uses reflection self-report (tier 3) only. Contradiction log entries for these agents can only open at Level 1 when tier 2 artifact evidence exists (rare for assistants). Most patterns from assistant agents will be tier 3 proposals only, which cannot be contradictions — only personal guidance candidates.

## Part 9: Startup gate v2

### Two-stage startup model

The gate cannot check for disposition artifacts that startup itself is supposed to produce. That is circular. The startup sequence is therefore split into two stages:

**Stage 1 — Mandatory preflight (Kvothe does the work):**
```
0. Compute and persist locked open-date:
   Write memory/synapse/.preflight-date
   Content: { "open_date": "{YYYY-MM-DD}", "open_date_yesterday": "{YYYY-MM-DD}", "session_candidate": "{open-date}-{HHMM}" }
   Where `session_candidate` is the anticipated Session ID — the open-date plus the HHMM at the moment Step 0 runs. This becomes the Session ID embedded in the session-open marker written by Stage 2.
   This file is the canonical source of the locked date before the session-open marker exists.
   startup-gate.sh reads this file to get the open-date to pass into all checks.
   shutdown-gate.sh deletes this file on clean close (same time as session-open marker).
1. Read memory/synapse/proposals/{open-date-yesterday}.md
2. Write memory/synapse/proposals/{open-date-yesterday}.reviewed (all candidates dispositioned)
3. Read memory/synapse/contradiction-log.md
4. Add review_note to any OPEN entries without one
5. Read memory/chronicles/{open-date}.md (create if missing)
6. Read memory/ACTIVE.md
```

Stage 1 produces the artifacts (including `.preflight-date`). Stage 2 reads `.preflight-date` to get the locked open-date and verifies the rest.

**Stage 2 — Gate verification (script runs, checks artifacts):**

```bash
./scripts/startup-gate.sh
```

**CHECK 1: Chronicle for locked open-date**
Pass: `memory/chronicles/{open-date}.md` exists.

**CHECK 2: ACTIVE.md freshness**
Pass: modified within last 8 hours.

**CHECK 3: Proposals dispositioned**
Pass: `memory/synapse/proposals/{open-date-yesterday}.reviewed` exists AND has a disposition line for every candidate.
Trivial pass: no proposals file for `{open-date-yesterday}`.
Fail: proposals exist but .reviewed file missing or incomplete.

**CHECK 4: Contradiction review_notes**
Pass: no OPEN contradiction entries are missing a review_note field.
Fail: any OPEN entry without review_note.

**CHECK 5: MEMORY.md freshness**
Warning (not block): stale >3 days. Flagged for session end.

**CHECK 6: NOTES.md overdue items**
Fail: any past-due review dates.

**CHECK 7: No orphaned session markers (any date)**
Checks all `.session-open-{session_candidate}` markers across all dates. For each marker, the embedded `session_candidate` value is the Session ID — checks whether a session-log entry with that `Session ID` exists. Matching is by Session ID — not by date. Two sessions on the same day have different `session_candidate` values and are checked independently.

Severity policy (one consistent ladder):
- Orphaned marker <24h old: **warning** — flagged, does not block startup. shutdown-gate.sh will refuse close while any unresolved orphan warning marker exists (excluding the current session's own open marker). The session can open, but cannot close cleanly until the warning is resolved. **Resolution:** write a manual close entry to session-log (see format below), then delete all three artifacts matching the orphaned Session ID: (1) the `.session-open-{session-id}` marker, (2) `.preflight-date` if its `session_candidate` exactly matches the orphaned Session ID, (3) `.session-type-{session-id}` if it exists. Do not delete `.preflight-date` on `open_date` match alone. Same resolution as the ≥24h path.
- Orphaned marker ≥24h old: **hard fail** — blocks startup. **Resolution:** write a manual close entry to session-log (see format below), then delete all three artifacts matching the orphaned Session ID: (1) the `.session-open-{session-id}` marker, (2) `.preflight-date` if its `session_candidate` exactly matches, (3) `.session-type-{session-id}` if it exists.

**Manual close entry format** (both resolution paths):

Manual close entries are an **explicit session-log schema exception**. The `Session type` field uses `unknown` rather than `active | heartbeat-only`, and operational fields use `unknown` rather than real values. This is intentional — parsers and watchdog logic must treat `unknown` as a valid field value in manual-close entries. The authoritative key is still `Session ID`; the `unknown` values signal that the session did not complete normally rather than representing missing data.

```markdown
---
## Session — {open-date} — manual close
Session ID: {session_candidate} ← must match the orphaned marker's session_candidate exactly
Session type: unknown — session did not complete normally
- Archive review (GENERAL_LEARNINGS): unknown
- Archive review (NEEMCLAW_LEARNINGS): unknown
- Kvothe reflection: unknown — manually closed
- Proposals reviewed: unknown
- Kilvin spawns (this session): unknown
```
The `Session ID` field must carry the orphaned marker's exact ID so watchdog Session ID matching finds it. Ad hoc text notes are not sufficient — they will not satisfy the session-ID lookup and the orphan will continue to be flagged.

No "any orphaned marker = fail" catch-all. Age determines when the block occurs (startup vs shutdown). Resolution is the same for both.

**On pass:** Write session-open marker `memory/synapse/.session-open-{session_candidate}` where `{session_candidate}` is the exact value already stored in `.preflight-date` — not a freshly computed HHMM from the wall clock at pass time. This guarantees the session-open marker's embedded Session ID matches the `session_candidate` written by Stage 1 step 0.

**Kilvin per-spawn tracking (checked at shutdown, not startup):**
`kilvin-spawn.sh` writes a spawn marker on every invocation:
`memory/synapse/.kilvin-spawn-{open-date}-{HHMM}`
Content: `{ "spawned_at": "{ISO timestamp}", "session_id": "{open-date}-{HHMM-of-current-session}", "task": "{task name from prompt first line}" }`

The `session_id` field is the current Kvothe session's ID — the `session_candidate` value from `.preflight-date`, which is also the filename suffix of the active `.session-open-{session_candidate}` marker.

shutdown-gate.sh counts `.kilvin-spawn-{open-date}-*` markers and counts `## Reflection —` entries in `memory/reflections/kilvin/{open-date}.md`.
If marker count > reflection entry count: gate fails. Missing reflections must be resolved (write backup entries labeled "backup — spawn did not complete reflection step") before /new.

**Output on pass:**
```
✅ Startup gate clear — {timestamp}
Session ID: {session_candidate}
Ready.
```

**Output on fail:**
```
🔒 Startup gate — {N} items need attention:
[each failed check with exact action required]
Do not proceed. Re-run: ./scripts/startup-gate.sh
```

**What the gate proves vs does not prove:**
- ✅ Proves: disposition decisions written for each proposal
- ✅ Proves: contradiction review_notes present
- ✅ Proves: prior sessions closed cleanly
- ❌ Does not prove: decisions were good or thorough
That limit is accepted. Evidence of engagement, not proof of comprehension.

```

Content: "active" if real work happened. "heartbeat-only" if no real work, no decisions, no tasks.
This file is read by shutdown-gate.sh. Do not skip. Gate fails without it.

BEFORE STEP 1 — LEARNINGS REVIEW
Read `memory/chronicles/{open-date}.md`. Apply trigger test for each archive file.

If no trigger fired → write nothing to archive.
Note: session-log.md entry is written by shutdown-gate.sh (not by Kvothe directly — see below).

AFTER STEP 3 (Chronicle) — KVOTHE REFLECTION
Conditional on session type:
- Active session (real work happened): write memory/reflections/kvothe/{open-date}.md using Strategic format. Append if file already exists. Required.
- Heartbeat-only session (no real work, no decisions, no tasks): skip reflection. Log "heartbeat-only session — skipped" in session-log entry.

**Session type is declared by Kvothe at shutdown**, not inferred. The canonical handoff mechanism:

1. Kvothe writes a session-type declaration file as the first shutdown step:
   `memory/synapse/.session-type-{session-id}` with content: `active` or `heartbeat-only`
2. `shutdown-gate.sh` reads this file to get the declaration
3. After writing the session-log entry, shutdown-gate.sh deletes the declaration file
4. If the declaration file is missing when the gate runs: gate fails — Kvothe must write it before proceeding

This gives the gate a single unambiguous artifact to read. No CLI args, no prompts, no env vars — one file, one value.

**Per-session reflection enforcement:** shutdown-gate.sh evaluates:
- **Prior active closes** = count of `Session type: active` entries already written in session-log.md whose Session ID shares the current open-date (i.e., same `YYYY-MM-DD` prefix) — these are earlier sessions from the same day
- **Current session declaration** = content of `memory/synapse/.session-type-{session-id}` written by Kvothe as the first shutdown step (`active` or `heartbeat-only`)
- **Total expected reflections** = prior active closes + (1 if current declaration is "active", 0 if "heartbeat-only")
- **Actual reflections** = count of `## Reflection —` entries in `memory/reflections/kvothe/{open-date}.md`

If total expected > actual: gate fails — append reflection entry for current session before /new.
The current session's close entry is NOT yet in session-log.md at check time. The gate uses the declaration to account for it. This prevents the current session from slipping through on same-day multi-session runs.

AFTER STEP 8 (Verify) — KILVIN CHECKS (if any Kilvin spawn occurred on this open-date)
This check is **spawn-day scoped**, not session-scoped. It runs in every session that closes on a day where at least one Kilvin spawn marker exists — even if this particular session did not spawn Kilvin.

Verify marker count vs reflection-entry count (open-date cumulative):
  - Count .kilvin-spawn-{open-date}-* markers in memory/synapse/
  - Count ## Reflection — entries in memory/reflections/kilvin/{open-date}.md
  - If marker count > reflection count: write backup entries for missing spawns before proceeding

**KILVIN_MEMORY.md update — spawn-day requirement:**
Update workspace-kilvin/KILVIN_MEMORY.md once per spawn day — version, recent builds, open issues. Write update stamp: `*Last updated: {open-date} (spawn: {HHMM})*`. The requirement is date-only: shutdown-gate verifies the stamp date matches the spawn open-date. It does NOT verify which session wrote the stamp or which HHMM value appears. Any session closing on that open-date may write the stamp — the first session to do so satisfies the requirement for the day. mtime is not used.

**Session-log field is session-local:** The `Kilvin spawns (this session)` field logged in session-log.md records only the spawns from *this* session. Session-local count is computed by reading the `session_id` field from each `.kilvin-spawn-{open-date}-*` marker and comparing it to the current session's ID. Markers whose `session_id` matches the current session belong to this session. This uses the authoritative Session ID key — the same key used for all other session matching in this spec. Enforcement (total marker count vs total reflection count) is open-date cumulative and may exceed what any single session logs — that is intentional. The log describes what happened in this session; enforcement ensures nothing is missing across the whole day.

THEN: run shutdown-gate.sh
```

**Session-log ownership:** `shutdown-gate.sh` is the single canonical writer of session-log.md entries. - Add deprecation headers to: learnings-index.md, kilvin-patterns.md, spec-quality.md
- Add archive headers to: GENERAL_LEARNINGS.md, NEEMCLAW_LEARNINGS.md

### Phase 2 — Scripts
- Write `scripts/startup-gate.sh` v2:
  - **Reads locked open-date from `memory/synapse/.preflight-date`** (written by Stage 1 step 0). Does NOT recompute or re-lock the date — uses the value from the file. This ensures Stage 1 preflight and Stage 2 gate always operate on the same date even if the clock crosses midnight between them. If `.preflight-date` is missing when the gate runs: gate fails — Stage 1 was not completed.
  - Checks: chronicle exists (for the received open-date), ACTIVE.md fresh, .reviewed file complete, contradiction review_notes present, MEMORY.md freshness warning, NOTES.md overdue, orphaned session markers (age-based severity: <24h warning, ≥24h hard fail)
  - On pass: writes `.session-open-{session_candidate}` using the exact `session_candidate` value from `.preflight-date` — not a fresh wall-clock HHMM
- Update `scripts/session-watchdog.sh`:
  - For each `.session-open-{session_candidate}` marker: the embedded `session_candidate` is the Session ID; checks whether a session-log entry with that Session ID exists. Matching by Session ID — not by date. Two sessions on the same day are checked independently.
  - Alerts on inactivity: no writes to `ACTIVE.md`, `memory/reflections/`, or `memory/guidance/` within last 4h, AND no matching close entry for this Session ID. Exempt: `synapse/proposals/*.reviewed` and `synapse/contradiction-log.md`. No other paths in alert set. Not on raw session duration.
  - Preflight writes exempt: writes to `synapse/proposals/*.reviewed` and `synapse/contradiction-log.md` without a .session-open marker do NOT trigger skipped-startup alert
- Update `scripts/shutdown-gate.sh`:
  - Reads `.session-type-{session-id}` declaration file for active vs heartbeat-only
  - Writes single session-log entry (sole writer)
  - Checks Kvothe reflection count (active closes vs reflection entries, per open-date)
  - Checks KILVIN_MEMORY.md stamp date matches spawn open-date (on spawn days)
  - Refuses close while unresolved orphan warning markers exist (excluding current session's own marker)
- Update `SHUTDOWN_CHECKLIST.md`
- Update `AGENTS.md` startup sequence (steps point to new paths)
- Update `KILVIN_SPAWN_PROTOCOL.md` (guidance/kilvin.md + guidance/shared.md; reflection append step; REFLECTION_WRITTEN required)

### Phase 3 — Synthesis cron
- Read `memory/reflections/*/` using mtime-based catch-up (file-mtimes.json) — not a fixed yesterday window
- For each reflection file read, also read its open-date chronicle (chronicle coupling rule — see Part 8)
- Dedupe via processed-registry.jsonl using 4-tuple ID: `{agent}::{open-date}::{N}::{HH:MM}`
- Evidence tier classification in proposals
- Contradiction-log update: idempotent — key is `agent + conflicting guidance key + evidence reference`; a match on all three skips; a match on agent + evidence alone but different guidance key is a new legitimate entry
- Commit order: proposals + chronicle-reflections narrative → contradiction-log → registry → mtimes → checkpoint
- Write proposals to `memory/synapse/proposals/{date}.md`; write synthesis narrative to `memory/synapse/chronicle-reflections/{date}.md`
- Checkpoint with agent list, missing agents, low-quality reflections flagged (≥5 sample before rate threshold applies)
- Session event to Kvothe on completion

### Phase 4 — Agent SOUL.md updates
- All 11 in-scope agents: reflection format, trigger, startup reads, guidance path

### Phase 5 — Validation (7 days)
- Daily: reflection rate per agent, proposal evidence tier distribution, contradiction-log accuracy, gate compliance, session-log completeness
- After 7 days: review policy defaults (thresholds, confidence floors) against actual data

## Part 13: Failure modes

| Failure | Impact | Detection | Recovery |
|---|---|---|---|
| Kvothe skips startup gate | Stale context, no session-open marker | Watchdog (**best-effort** — not exhaustive): writes to `ACTIVE.md`, `memory/reflections/`, or `memory/guidance/` without a `.session-open` marker, sustained 30+ min. Exempt: `synapse/proposals/*.reviewed` and `synapse/contradiction-log.md` (valid Stage 1 preflight artifacts). No other paths in alert set. An ungated session that only writes to chronicles or spec/task files will not trigger this alert — the narrow signal set is intentional to avoid false positives, not a guarantee of full detection. | Alert Rai. Retroactive gate run. Two distinct paths: **(A) Stage 1 never happened** — open-date was never locked; Kvothe manually determines the correct open-date (use timestamp of first work file written this session), runs Stage 1 preflight, then runs the gate. **(B) Stage 1 happened but startup-gate.sh was skipped** — open-date was already locked; read it from `memory/synapse/.preflight-date` (written by Stage 1 step 0). Run `startup-gate.sh` — it will read `.preflight-date` automatically. In both cases: ensure session-open marker exists before closing. |
| .reviewed file incomplete or missing | Gate blocks next session startup | startup-gate.sh CHECK 3: verifies `.reviewed` file exists and has a disposition line for every candidate from proposals file | Write missing dispositions. If no proposals file exists for `{open-date-yesterday}`: trivial pass, no action needed. |
| Orphaned session-open marker | Watchdog false-alarm; shutdown blocked (if <24h warning marker unresolved) | Watchdog: `.session-open-{session_candidate}` marker exists but no session-log entry with `Session ID: {session_candidate}` found, AND no observable workspace writes in last 4h | Write a full manual close entry using the orphaned `session_candidate` as Session ID (see CHECK 7 format). Delete all three matching artifacts: `.session-open-{session_candidate}`, `.preflight-date` (if `session_candidate` exact match), `.session-type-{session_candidate}` (if exists). Use manual-close schema (Part 9 CHECK 7) — ad hoc text not sufficient. |
| Agent doesn't reflect | No proposals for that agent that day | Synthesis checkpoint lists missing agents; missing count reported | Synthesis runs on available data. No recovery needed unless pattern repeats. |
| Synthesis fails mid-run | Partial outputs; processed-registry not updated if failure before step 4; file-mtimes not updated if failure before step 5 | No complete checkpoint for that run date; proposals file may be partial or absent | Next run: mtime catch-up re-reads all files not recorded in file-mtimes.json. Processed-registry dedupes any IDs already committed. Proposals may be written twice — Kvothe discards duplicates at review. contradiction-log is idempotent (agent + guidance key + evidence reference triple) — no duplicate openings. Source data always preserved. |
| Guidance file corrupted | Agents read wrong guidance | Kvothe flags on startup read | Rebuild from chronicle + archive. If a personal guidance key is unrecoverable, discard and rebuild from agent's reflection history. |
| Contradiction opened from tier 3 evidence only | Noisy contradiction-log | Proposal tagged tier 3 — cannot create a contradiction entry (only tier 1/2 can open Level 1) | Kvothe discards at review. No log entry created. |
| Seeding GS from corrupt learnings-index | Corrupt rules in shared guidance | GS entry missing archive reference or unverifiable rule source | Discard during Phase 1 audit. Every GS entry must have a verifiable archive narrative. Corrupt entries are not carried forward. |
